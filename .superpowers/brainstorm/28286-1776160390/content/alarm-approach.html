Flow Node — Alarm Design

How should alarms work in the node?

perf-monitor
output 1 — metrics
msg.payload = { cpu: 12.4, memory: { used: 512MB, pct: 63 }, eventLoopLag: 1.2 }
msg.topic = "metrics"
→ Downstream: Switch node checks cpu > 80 → alarm branch

A — Single output, alarms via Switch

Node outputs metrics. Developers wire a Switch node downstream to branch on thresholds. Simple node, full flexibility — any field, any condition, any logic.

Pros

  • Simpler node config
  • Full Node-RED composability
  • No duplicate logic

Cons

  • More wiring for users
  • Threshold config spread across nodes
perf-monitor
output 1 — metrics
output 2 — alarm 🔴
Node editor: CPU threshold %
Node editor: Mem threshold %
→ Output 2 fires: { alarm: "cpu", value: 94.2, threshold: 80 }

B — Dual output with built-in thresholds

Node has a second output port that fires when configured thresholds are breached. CPU, memory, event loop lag thresholds set in the node editor. Alarm msg carries context (which metric, value, threshold).

Pros

  • All-in-one, less wiring
  • Alarm context bundled
  • Discoverable for new users

Cons

  • More complex node
  • Limited to pre-defined metrics
