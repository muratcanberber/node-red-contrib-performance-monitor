A — Shared Store Hub (recommended)

All three subsystems consume MetricsStore. AnomalyDetector subscribes to store events, fires RED runtime events + triggers flow node alarm output. Report page queries store HTTP API. Flow node reads live sample.

Pros

  • Single source of truth
  • No duplication
  • Anomaly detector reuses persistence

Cons

  • All three ship together
Flow node ← own polling
Report page ← own HTTP fetch
Anomaly ← own metric sampling

Three separate collectors, no shared state

B — Independent modules

Each subsystem manages its own data pipeline independently. More isolated but triples sampling overhead.

Pros

  • Fully decoupled

Cons

  • 3× CPU/sampling overhead
  • Data diverges between views
  • Much more code
MetricsStore → HTTP API
All clients poll REST endpoints
No SSE, no event bus

REST-only, simpler but less real-time

C — REST-only, no event bus

Everything polls HTTP endpoints. Simpler but anomaly detection becomes polling-based (detects later), report page can't get sub-second updates.

Pros

  • Simplest integration

Cons

  • Delayed anomaly detection
  • No live streaming to report page
